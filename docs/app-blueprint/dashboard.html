Strummy · Application Blueprint

The Main Blueprint: Architecture & Status

This is the primary source of truth for the Strummy application. It details the application's overall vision, architecture, the three primary roles, and the real-time status of every feature across all blueprint domains. The data below is a comprehensive representation of what each role can reach and do, verified against the actual codebase and live database snapshot.

Application Overview

Strummy is a student management system for guitar teachers — a single-teacher CRM/LMS run by its owner for his own studio. The teacher schedules lessons, curates a song catalog, assigns repertoire and homework; students log in to see their lessons, songs, assignments, and log practice. Around that core sit an AI assistant, a notification engine, Google Calendar sync, Spotify enrichment, a theory-course LMS, and a video/social content pipeline.

Strategic Frame

The current strategic frame is a trust pass, not a feature pass. The immediate goal is onboarding real students onto a clean self-hosted production stack. The student-facing surface must be honest — no placeholder features — and the working core loop must be reliable. New student-facing features (gamification, tempo ladder, SRS review) are slated for v1.1, built after real usage.

Roles & Access Model

Three roles enforced via Supabase RLS: Admin, Teacher, and Student (an is_parent flag exists but the parent experience is unbuilt). The owner is currently the only teacher, and admin and teacher views coincide in practice. Students never see each other's data — proven by the RLS cross-read E2E tests.

CRUD rule of thumb: Admin/Teacher — full CRUD on all rows of every domain table. Student — reads/writes only own rows; teaching artifacts (songs, lessons, assignments) are read-only, with a few explicit self-service writes (e.g., advance own assignment status, log practice, repertoire self-rating).

How a request flows

RLS is the security enforcement point (ADR-0001) — every layer above it is convenience and UX, never the boundary. The app is RSC-first: pages fetch on the server through lib/services/*-queries.ts, mutations go through Server Actions with Zod at the boundary. The REST surface under app/api/ serves external/API-key consumers, widgets, cron, and webhooks — not page rendering (see API & jobs tab).

Three-layer auth

Layer Where Catches
1 · Edge proxy.ts (root — Next 16 middleware) Unauthenticated on /dashboard/*; deactivated accounts (forced sign-out); non-admin on /dashboard/admin/* — the only place the admin gate exists; security headers (CSP allowlists Supabase, Google OAuth, Sentry, PostHog; frame-src 'none'; no unsafe-eval)
2 · Layout app/dashboard/layout.tsx Missing session → /sign-in; role-less user → /onboarding (covers password sign-in, which the OAuth callback misses)
3 · Page getUserWithRolesSSR() in each page.tsx Teacher-vs-student branching, redirect() for the wrong role, notFound() on non-owned records — the inconsistent layer (see banner)
Guard gaps (verified 2026-07-19) Pages that skip layer 3 and rely on proxy + RLS alone: /ai (AI dev harness outside /dashboard — gets neither the admin gate nor the layout gate), /dashboard/ai/history, /dashboard/skills/chord-quiz, and the theory chapter reader /dashboard/theory/[courseId]/[lessonId] (only notFound()). And proxy.ts fails open: if getSupabaseConfig() throws (missing env), requests pass with no auth check and no security headers.

Key mechanisms

Roles & RLS

is_admin() · teacher_teaches_student()

Roles are boolean flags on profiles (user_roles is legacy — dropped by migration 20260323120000, though still live on StudentManager). Policies use SECURITY DEFINER helpers (is_admin(), current_user_roles()…) to break RLS recursion — fixed five separate times. The teacher↔student relation is derived from lessons: teacher_teaches_student() checks for a non-deleted lesson between the pair; there is no membership table.

Data fetching

RSC-first · Server Actions

Async server component → lib/services/*-queries.ts → cookie-bound client from lib/supabase/server. Mutations: 'use server' action → Zod safeParse → revalidatePath. TanStack Query survives in 11 files (mutation hooks, 2 stat hooks, production tab). AI streaming is its own lane (useAIStream → SSE).

Dual DB routing

local ⇄ remote Supabase

lib/supabase/config.ts resolves local vs remote from env; preference via X-Database-Preference header or sb-provider-preference cookie. next.config.ts TCP-probes the local stack (1s timeout) at boot and deletes the LOCAL_* vars in dev when unreachable — silent remote fallback.

Notification engine

triggers → queue → SMTP + in-app

DB triggers (tr_notify_lesson_*, tr_notify_song_mastery…) enqueue into notification_queue; a 6-hourly GitHub-Actions cron drains it (get_pending_notifications() batch dequeue) through Gmail SMTP and in-app-notification-service. HMAC-signed unsubscribe tokens; bounce handler + get_bounce_stats(); per-type user preferences.

AI system

provider factory · 11 agents

AI_PROVIDER = auto | openrouter | ollama (auto prefers local, falls back to cloud, 3 retries). Agents registered in lib/ai/registry with prompt-injection validation and sensitive-data modes; role-based rate limits (admin 100/min · teacher 50 · student 20 · anon 5); queue: 2 concurrent, 5 waiting, 60s timeout; token-estimation drives the progress UI.

Navigation truth

menuConfig.ts

One config feeds desktop sidebar and mobile nav. CORE_LOOP_HIDDEN_ITEMS (13 ids) hides everything outside the core loop for every role — nav-only, routes stay URL-reachable. Admin and teacher nav are byte-identical; there is no admin-only nav entry. DEMO_HIDDEN_ITEMS is dead code (a strict subset of the core-loop list).

Database internals

~78 functions · ~72 triggers

Monthly-partitioned audit_log (auto-created partitions) with jsonb_diff history rows; practice→repertoire aggregation triggers; lesson auto-numbering; materialized-view refreshers (refresh_dashboard_stats, song popularity/engagement, teacher performance). ⚠ Three competing updated_at trigger conventions and mixed trigger prefixes (tr_/trg_/trigger_).

Observability

Sentry · PostHog · Pino

Sentry wired for all three runtimes via instrumentation.ts + withSentryConfig, replay on, traces 1.0. PostHog identifies users with role flags. Pino writes structured logs, surfaced at /dashboard/logs (system_logs); lib/observability/home-ops-log.mjs bridges to the home-ops stack.

Where do I add X? (recipes)

A dashboard page

app/dashboard/<x>/page.tsx

Guard it yourself with getUserWithRolesSSR() + redirect()/notFound() — the proxy only covers auth + /dashboard/admin. Add the nav item in components/navigation/menuConfig.ts; anything whose id lands in CORE_LOOP_HIDDEN_ITEMS stays URL-only for every role. Desktop sidebar and mobile nav both read this config.

A mutation

app/actions/<domain>.ts

'use server' + Zod schema from /schemas + safeParse at the top + revalidatePath on success. Colocate a unit test. Don't add a REST route for an in-app mutation — Server Actions are the house path.

A table

supabase/migrations/

Timestamped migration (YYYYMMDDHHMMSS_name.sql — never the legacy 0NN_ style). Enable RLS immediately; write policies with is_admin() / teacher_teaches_student(); add an updated_at trigger (prefer set_updated_at). Regenerate root database.types.ts (npx supabase gen types typescript) and add an RLS suite in lib/testing/rls/__tests__/.

An API route

app/api/<area>/route.ts

Wrap with withApiAuth({ requiredRole }) from lib/auth/withApiAuth.ts — not raw getUser() (that idiom is legacy, ~40 routes deep). Response envelope from lib/api/response.ts; never a bare data field. Add an integration test; regenerate npm run audit:routes.

A cron job

app/api/cron/<name>/route.ts

GET handler behind verifyCronSecret (500 = secret unset, 401 = mismatch). Schedule it in vercel.json — or GH Actions cron-jobs.yml if sub-daily (mind the Actions bill: 6-hourly, not */15). Update lib/health/cron-registry.ts so the health UI tells the truth, and add a test in __tests__/app/api/cron/.

An AI agent

lib/ai/agents/

Write the agent spec, register it in lib/ai/registry (validation + sensitive-data mode), expose a typed wrapper in app/actions/ai/<domain>.ts. Rate limiting, queueing, retries, and streaming come from the shared plumbing; add a token-estimation entry so the progress bar works.

A notification type

triggers → queue → template

Enqueue from a DB trigger (tr_notify_*) or service call into notification_queue; add the email template under lib/email/templates/ + in-app content in notification-in-app-content; register the per-type toggle in notification preferences. Delivery happens on the 6-hourly queue drain — nothing is instant.

An E2E journey

tests/e2e/<area>/

Use tests/fixtures/auth.fixture.ts + timestamped data matching the cleanup patterns in tests/helpers/cleanup.ts (or it leaks). Max 10 E2E per feature — push scenarios down to integration. Tag it so test:pw:* runners pick it up.
