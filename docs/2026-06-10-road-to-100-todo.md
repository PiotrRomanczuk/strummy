# Road to 100% — Specific TODO List

**Date**: 2026-06-10
**Author**: Claude
**Scope**: Phased plan to take Strummy from current state to "100%" — no Coming-soon pages, no flows that strand users, no silent failures.

Companion to `docs/2026-06-10-backend-audit.md` (the audit findings this TODO works from). Every item names the files to touch and the acceptance criterion ("done when"). Ordered by dependency: each phase assumes the previous one. Estimates are working-effort, solo.

## Phase 0 — Restore truth (CI signal + production bugs) · ~3–4 days

**0.1 Fix Jest worktree exclusion** · 30 min

- File: `jest.config.ts` → `testPathIgnorePatterns`
- Change the bare `/.claude/` pattern to `<rootDir>/.claude/` so tests can run from
  `.claude/worktrees/*` and the ~20 silently-excluded component tests run again.
- Done when: `npm test` from a worktree runs the same suite count as from the repo root.

**0.2 Consolidate bearer auth** · 1–2 days

- Delete `lib/bearer-auth.ts`; migrate every route importing `authenticateWithBearerToken()`
  to `withApiAuth()` (`lib/auth/withApiAuth.ts`), which already handles `gcrm_` keys +
  session cookies via `lib/auth/api-auth.ts`.
- Known routes: `app/api/external/songs/route.ts`, `app/api/widget/dashboard/route.ts`,
  `app/api/widget/admin/route.ts`, `app/api/song/route.ts`; grep for the rest.
- Done when: `rg "bearer-auth"` returns nothing and `npm run test:bruno` passes with a real
  `gcrm_` key against preview.

**0.3 Fix the 500-ing crons** · 1 day

- Routes: `app/api/cron/weekly-insights`, `app/api/cron/weekly-digest`,
  `app/api/cron/cleanup-auth-events` (per `docs/2026-06-10-backend-audit.md`).
- Reproduce locally with `CRON_SECRET` set; likely causes per audit: missing RPC/migration
  drift and null-data assumptions in `lib/services/weekly-insights.ts`.
- Done when: each route returns 200 in prod (check Vercel logs after next scheduled run) and
  each has a regression test.

**0.4 Reconcile migration drift** · ½–1 day

- `supabase migration list` against remote (88 applied) vs `supabase/migrations/` (154 files);
  repair history or archive orphans/backups out of the migrations dir.
- Verify `check_auth_rate_limit` RPC (migration `036_auth_rate_limits.sql`) exists remotely —
  sign-up rate limiting silently depends on it.
- Done when: `supabase db diff` against remote is empty and migration list matches 1:1.

**0.5 Audit the `SECURITY DEFINER` view** · 2 h

- From `docs/2026-06-10-backend-audit.md`: one view bypasses RLS. Either add an explicit role
  check inside it or convert to `SECURITY INVOKER`.
- Done when: querying the view as a student returns only student-visible rows (add an RLS
  test in Phase 4).

## Phase 1 — Finish the CRUD UI (the 12 stubs) · ~2–3 weeks

All backends exist; this is UI wiring. Use `components/*/editorial/*` (the routed generation).

**1.1 Songs list** — `app/dashboard/songs/page.tsx`

- Wire to `GET /api/song` (filters: level, key, author, search; pagination already in API).
- Done when: teacher sees all songs, student sees only assigned (RLS already enforces).

**1.2 Songs edit** — `app/dashboard/songs/[id]/edit/page.tsx`

- Mount the already-built `SongEditFormEditorial`. Smallest item on this list.

**1.3 Lesson create** — `app/dashboard/lessons/new/page.tsx`

- Form: student picker (reuse inline shadow-create via `matchStudentByEmail()`), date/time,
  title, notes → `POST /api/lessons`. Calendar push already happens in the handler
  (`syncLessonCreation`).
- Done when: creating a lesson with a Google-connected teacher produces a calendar event.

**1.4 Lesson edit** — `app/dashboard/lessons/[id]/edit/page.tsx`

- Fields: title, scheduled_at, status, notes → `PUT /api/lessons/[id]` (syncs to Google).

**1.5 Assignments create/detail/edit** — `app/dashboard/assignments/new`, `[id]`, `[id]/edit`

- Create: student + optional lesson/song pickers → `POST /api/assignments` (validation and
  student notification already in `createAssignmentHandler`).
- Detail: render assignment + status timeline; student sees status control
  (`updateAssignmentStatus` action — status-only, already enforced server-side).
- Done when: teacher creates → student sees it + gets in-app notification → student marks
  in_progress/completed → teacher sees the change.

**1.6 Users list/detail/edit** — `app/dashboard/users/*`

- List: `GET /api/users` (search, role, studentStatus filters). Detail: profile + lessons +
  assignments summary. Edit: role flags and student_status admin-only (API already guards).
- Include shadow badge + "Invite" button placeholder (wired in Phase 2.1).

**1.7 Repertoire page** — `app/dashboard/repertoire/page.tsx`

- Add thin server actions first (recommended over raw RLS queries):
  `app/actions/repertoire.ts` → `getRepertoire(studentId?)`, `updateRepertoireEntry()`
  (teacher: all fields; student: notes + difficulty only — mirror RLS policies
  `sr_update_own_notes` / `sr_update_admin_teacher`).
- Student view: own songs with status/notes/practice stats. Teacher view: per-student.

**1.8 Practice history** — `app/dashboard/practice/page.tsx`

- New read action `getPracticeSessions()` in `app/actions/practice.ts` (none exists — students
  currently log into a void).
- Decide: keep sessions immutable but add a same-day "undo" delete, or stay strict.

**1.9 Student profile self-edit** — settings or `/dashboard/profile`

- Form for full_name/phone/avatar → existing `PATCH /api/users/[id]` self-update path.

**1.10 Multi-role rendering sweep** · ½ day

- `rg "isStudent && !isTeacher"` and similar; replace single-role assumptions with
  highest-role-wins or explicit role switch. You (admin+teacher) are the main victim.

## Phase 2 — Shadow users & auth completion · ~1 week

**2.1 Shadow invite flow** (ADR 0002) · 1–2 days — _highest-value item in the project_

- UI: "Invite" dialog on user detail (1.6): input real email → `PATCH /api/users` sets
  `invite_email` → call `supabase.auth.admin.inviteUserByEmail(invite_email)` (admin client)
  → log `shadow_invite_sent` via `lib/auth/auth-event-logger.ts`.
- Linking already works (`handle_new_user()` matches `invite_email`).
- Done when: invite → student clicks email → signs up → lessons/assignments appear on their
  account, shadow gone, `shadow_link_completed` logged.

**2.2 Deliverable-email chokepoint** · 1 day

- New `lib/email/recipient.ts` → `getDeliverableEmail(profile)`: shadows return
  `invite_email ?? null`; null ⇒ skip send + write `skipped_shadow` row to `notification_log`.
- Route ALL senders through it: `lib/services/send-lesson-email.ts`,
  `send-reminder-email.ts`, notification queue processor, AI email drafts.
- Done when: no email is ever addressed to `shadow_*@placeholder.com` (assert in a unit test).

**2.3 Calendar reconciliation on link** · ½ day

- On `shadow_link_completed`, enqueue task; processor patches attendee email on future events
  with `google_event_id` for that student.

**2.4 MFA: finish or remove** · 1 day either way

- Sign-in already returns `{ mfaRequired, factorId }` and then dead-ends.
- Finish: challenge/verify screen calling `supabase.auth.mfa.challenge()` + `verify()`.
- Or remove the branch until needed. **Decide — don't leave the dead end.**

**2.5 Google OAuth sign-in** · ½ day

- `/api/oauth2/callback` only stores integration tokens. Either implement
  `signInWithOAuth({ provider: 'google' })` properly (separate callback from the
  integrations one) or remove the sign-in affordance. Integrations flow stays as-is.

**2.6 Admin lockout visibility** · ½ day

- Admin widget listing profiles where `locked_until > now()` + unlock button (clears
  `failed_login_attempts`/`locked_until`).

**2.7 Stale-shadow cleanup cron** (optional) · ½ day

- ADR 0002: archive shadows with no invite_email and no lesson in 90 days.

## Suggested sequence (solo, ~6–8 weeks to "100%")

| Week | Focus                                                                 |
| ---- | --------------------------------------------------------------------- |
| 1    | Phase 0 entirely (signal + prod bugs) + 1.1/1.2 quick wins            |
| 2–3  | Phase 1 CRUD UI (lessons → assignments → users → repertoire/practice) |
| 4    | Phase 2 (invite flow + email chokepoint first)                        |

Definition of "100%": no Coming-soon page reachable from nav · no flow that strands a user
(MFA, shadow invite) · no silent failure (crons, shadow emails) · every core table RLS-tested
· PR CI runs smoke E2E and a blocking coverage gate.
