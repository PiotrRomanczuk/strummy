Full Student Intake Form

Date: 2026-07-23 · Branch: feature/student-form-full-fields · Scope: Expand the editorial "Add student" form from invite-only to the full mockup (Identity / Contact / Schedule / Billing), end to end.

1. What was built

The Add student form (/dashboard/users/new) was rebuilt from a minimal first/last-name + invite-email form into the four-section design mockup, with a live sticky preview. The same four field sections are now reused on the Edit form (/dashboard/users/[id]/edit). New profile columns back every field.

  • I · Identity — Full name (required), Instrument, Level button-group (required), Start date, Avatar-colour swatches.
  • II · Contact — Student email (feeds the existing invite flow), Student phone, Parent/guardian name, Parent email.
  • III · Schedule — Recurring Day, Time, Duration.
  • IV · Billing — Rate per lesson, Billing cycle, Goals/notes (stored in the existing notes column).
  • Preview — coloured avatar + initials, name, level, lesson slot, and $rate/cycle.

Only Full name and Level are required to save (matching the mockup).

⚑ BILLING — product review needed. Rate-per-lesson (lesson_rate, numeric) and billing cycle (billing_cycle ∈ per_lesson / weekly / monthly) are a new product surface for Strummy, stored directly on the student profile and teacher-only. Confirm the billing model (per-student flat rate vs. per-lesson override, currency, invoicing) before this is surfaced to end users.

2. Data model

Migration supabase/migrations/20260723120200_profiles_student_fields.sql adds 11 columns to public.profiles (idempotent, applied + verified on StudentDevelopment): skill_level, instrument, start_date, avatar_color, parent_name, parent_email, lesson_day, lesson_time, lesson_duration_minutes, lesson_rate, billing_cycle. CHECK constraints guard skill_level, billing_cycle, positive duration, and non-negative rate. Goals/notes reuse the existing notes column; student email reuses invite_email.

3. How to test manually

Setup

  1. Apply the migration to your dev stack (already done on StudentDevelopment): 
    ssh uwh "docker exec -i supabase_db_StudentDevelopment psql -U postgres -d postgres" \
  < supabase/migrations/20260723120200_profiles_student_fields.sql
  2. npm run dev, sign in as admin (p.romanczuk@gmail.com / test123_admin).

Create flow

  1. Go to /dashboard/users/new. Confirm four numbered sections + a sticky Preview showing "New student".
  2. Click Add student with an empty name → inline "Full name is required." + no request sent.
  3. Type a full name → the preview name + avatar initials update live. Pick a different avatar colour → preview circle recolours. Switch the Level button → preview subtitle updates.
  4. Fill Student email, a rate (e.g. 65), pick a day/time → Preview "Lesson" + "Rate" update (e.g. Thu 4:00 PM, $65/mo).
  5. Click Add student → redirect to the new profile. Verify in the DB: 
    select full_name, skill_level, instrument, start_date, avatar_color, parent_email,
       lesson_day, lesson_time, lesson_duration_minutes, lesson_rate, billing_cycle, notes, invite_email
from profiles order by created_at desc limit 1;

Edit flow

  1. Open the new student's …/edit. Confirm the Identity/Contact/Schedule/Billing sections are pre-filled, plus a "V · ACCESS" roles/status/active card. (Student email is intentionally not shown here — it is governed by the invite button on the detail page.)
  2. Change instrument, level, rate, goals → Save changes → "✓ Saved". Reload → values persist.
  3. Toggle Student off → the intake sections collapse to a plain Display-name field; save still works.
