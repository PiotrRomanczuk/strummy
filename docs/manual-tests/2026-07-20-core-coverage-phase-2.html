What was built

The blueprint promised 100% unit coverage for the Assignments / Lessons / Songs / Students core. This change delivers it: 47 source files now sit at a perfect 100 lines / 100 branches / 100 functions / 100 statements, and each one is pinned by its own coverageThreshold entry in jest.config.ts so it cannot silently regress.

This is a test-and-tooling change. No user-visible feature changed, but three genuine bugs were found and fixed along the way — two of them are user-visible, and they are what most of the manual testing below is about.

Behaviour changes worth testing by hand

1. Shadow students created from an email no longer get a blank first name

app/actions/lesson-edit.helpers.ts used firstName ?? 'New'. String.split never returns an empty array, so that fallback could never fire. An email whose local part starts with a separator — e.g. .emma@example.com — split to ['', 'emma'], so the shadow profile was created with an empty first name. Changed to ||.

  1. Sign in as a teacher or admin (see CLAUDE.md dev credentials).
  2. Go to /dashboard/lessons/new.
  3. In the student field, instead of picking an existing student, enter a brand-new email whose local part begins with a dot: .emma.stone@example.com.
  4. Fill in a date/time and save the lesson.
  5. Expected: a new shadow student is created named “New emma stone” — not a name starting with a stray space.
  6. Also confirm the normal case still works: john.smith@example.com → “john smith”.

2. Undated assignments now sort newest-first

lib/services/assignment-list-params.ts mapped a missing due date to Infinity. Because the comparator subtracts, two undated rows produced Infinity - Infinity = NaN, which makes a comparator inconclusive — so the documented “then newest” tie-break never ran and undated assignments came out in whatever order Postgres returned.

  1. Sign in as a teacher.
  2. Create at least three assignments for the same student with no due date, a few minutes apart so their created_at differs.
  3. Open /dashboard/assignments with the default “needs attention” sort.
  4. Expected: all undated assignments appear after any dated ones, and among themselves the most recently created is first.
  5. Regression check: assignments with due dates must still be ordered overdue-first, then soonest-due.

3. Checklist sanitising (no change, but previously untested)

sanitizeChecklist is called by both the assignment-create and template-edit editorial screens, and had no test at all. Behaviour is unchanged — this is a confidence check.

  1. Go to /dashboard/assignments/new.
  2. Add checklist items with padding and blanks: "  Practice scales ", an empty row, and a whitespace-only row.
  3. Save, then reopen the assignment.
  4. Expected: one item, Practice scales, with no surrounding whitespace. Blank and whitespace-only rows are dropped.
  5. Repeat on a template at /dashboard/assignments/templates.

4. Google Calendar lesson import still works (code was deleted nearby)

app/actions/import-lessons.ts was deleted — it had zero callers and was superseded by lib/services/calendar-bulk-import.ts. Worth one pass to confirm the surviving path is untouched.

  1. Go to /dashboard/calendar (not linked in nav; navigate directly).
  2. Connect Google if needed, then run a historical sync over a small date range.
  3. Expected: the SSE progress stream reports per-event created / shadow_created / skipped, and lessons appear with a google_event_id. Re-running dedupes.

Prove the coverage lock actually bites

A coverageThreshold key that matches no file passes silently, so the gate was verified by deliberately breaking it:

  1. Add an unreachable branch to any locked file, e.g. in app/actions/lesson-edit.helpers.ts: 
    if (studentId === 'CANARY-NEVER-HAPPENS') return { ok: false, error: 'canary' };
  2. Run npm run test:coverage.
  3. Expected: exit code 1 and the message Jest: "./app/actions/lesson-edit.helpers.ts" coverage threshold for branches (100%) not met: 94.73%
  4. Revert. (This was run during development and behaved exactly as above.)
