LES-3 — "Repeat weekly" lesson creation

2026-07-19 · Tranche 4 debt item (docs/app-blueprint/02-lessons-calendar.md)
Feature shipped Production-relevant bug found + fixed

The roadmap describes LES-3 as "wire or drop" — generateRecurringLessons existed with zero callers, and the brief's recommended default was to build the smallest honest version (a "repeat weekly for N weeks" checkbox) rather than delete it, unless the owner declines. Wiring it up for real and testing it against the live local database surfaced two genuine, unrelated bugs in that dormant code, plus a third, much bigger one in the shared local dev database itself. All three are described below.

What was actually done

  • UI. LessonFormEditorial (create mode only) now has a "Repeat weekly" checkbox + a weeks selector (4/6/8/12, components/lessons/editorial/form/LessonForm.Recurring.tsx). Checking it and submitting calls generateRecurringLessons instead of createLessonAction; unchecked behaves exactly as before (verified by E2E). The submit logic was extracted into a new useLessonFormSubmit hook to keep the component under the file-size convention.
  • Requires an existing student. Recurring lessons need a resolved studentId (the underlying schema requires a UUID) — the "+ new student by email" path isn't supported for repeats. Checking the box with a new-student selection shows an inline error rather than silently failing.
  • Google Calendar sync. generateRecurringLessons now calls syncLessonCreation once per created lesson (best-effort, matching the single-lesson creation path in app/actions/lesson-edit.ts) — the roadmap brief's "Google-connected teacher gets 4 events" acceptance criterion. The original dormant code never did this at all.

Bugs found and fixed while verifying this end-to-end

Bug 1 Dead column reference.

The original (never-executed) generateRecurringLessons inserted a creator_user_id field that doesn't exist on the lessons table at all — confirmed via \d lessons against the live schema. Every invocation would have failed with a Postgres/PostgREST schema-cache error. Removed the field; the table has no such column and nothing else references it.

Bug 2 — the big one lesson_status enum casing had drifted on the shared local dev database.

Every migration that creates lesson_status (003_enums.sql, 20251107121500_create_enums.sql) defines uppercase values (SCHEDULED, IN_PROGRESS, COMPLETED, CANCELLED). So does every write-side consumer: the Zod schema (LessonStatusEnum), the status dropdown in LessonForm.Fields.tsx, lesson-edit.ts, and this new recurring code. But the actual live enum on the shared uwh-hosted local Supabase instance had drifted to lowercase (scheduled, in_progress, completed, cancelled) — confirmed directly via psql, and confirmed that no migration in this repo ever created lowercase values, so this wasn't an applied-but-undocumented migration; it was out-of-band drift. A couple of read/display call sites (lib/services/lessons-queries.ts's STATUS_LABELS/ STATUS_COLOURS, app/dashboard/lessons/page.tsx's STATUS_KEYS) had already been patched defensively to accept either case — a sign this was a known, partially-mitigated issue, never fixed at the root.

Net effect before this fix: every lesson-creation code path that writes an explicit SCHEDULED/etc. status — including plain single-lesson creation via the UI, not just this new recurring feature — was silently broken against this local dev database. It's why tests/e2e/teacher/lessons-crud.spec.ts's full create/edit/delete test has sat test.skip'd with a stale "Coming Soon stub" comment; that reason is outdated, but the test genuinely couldn't have passed against this drifted state either.

Fix: supabase/migrations/20260719000007_fix_lesson_status_enum_casing.sql — four ALTER TYPE lesson_status RENAME VALUE 'x' TO 'X' statements. RENAME VALUE is metadata-only: existing rows keep the same underlying enum OID, so this is zero-downtime and zero-data-loss. Applied directly to the local uwh database (all 41 existing lesson rows read back correctly afterward) and recorded in supabase_migrations.schema_migrations.

Scope note: this fix targets the local dev database only. Whether production (zmlluqqqwrfhygvpfqka) has the same drift is unverified — the Supabase MCP connection to that project timed out when I tried a read-only check, and the codebase has an explicit hard guard (lib/testing/rls/env.ts) refusing to ever run test seeding against it. If production turns out to have the same lowercase drift, the identical RENAME VALUE migration is the fix — but that must be verified and applied deliberately, not inferred from this.

Fallout, fixed Two other test files were writing lowercase status: 'scheduled' directly to seed lessons — they matched the drifted local DB at the time they were written (earlier in this same effort), so they needed updating once the enum was corrected:

  • tests/e2e/teacher/calendar-conflicts.spec.ts (1 occurrence)
  • lib/services/__tests__/calendar-sync-recurring.rls.test.ts (4 occurrences)

Both re-verified passing after the fix (see Automated verification below).

Found, not fixed (out of scope)

Three unrelated RLS suites (profiles.rls.test.ts, songs.rls.test.ts, core-tables.rls.test.ts) fail deterministically against the current shared local database — 6 failing assertions about cross-tenant profile/song visibility and a lesson-trends view. Checked whether this traces to the enum fix: it doesn't — the one lesson insert in profiles.rls.test.ts never sets an explicit status (relies on the column default either way), and the failing core-tables.rls.test.ts assertion is about a row being absent entirely, from a view whose JOIN/WHERE clause never references lesson_status (only its internal COUNT() FILTER subclauses do, which affects counts, not row presence). Most likely explanation: data pollution from the very large number of E2E/RLS test runs against this single shared dev database today (multiple parallel worktree agents plus many manual runs). Flagging for Piotr rather than chasing further — resolving it isn't part of LES-3 and risks a much larger detour into unrelated test-isolation hygiene.

How to manually test it

  1. Ensure the local Supabase tunnel is up (ssh -fN -L 54321:localhost:54321 uwh) and start the app: npm run build && npx next start -p 3000.
  2. Sign in as the teacher demo account and go to /dashboard/lessons/new.
  3. Pick an existing student, a title, and a date/time. Leave "Repeat weekly" unchecked, submit — behaves exactly as before (redirects to the new lesson's detail page).
  4. Go back to /dashboard/lessons/new. This time check "Repeat weekly", pick "4 weeks", submit.
  5. You should land back on /dashboard/lessons (no single lesson to view). Confirm 4 new lessons exist, each exactly 7 days apart from the picked date, with sequential lesson numbers for that teacher/student pair.
  6. Edge case: try checking "Repeat weekly" while selecting "+ New student by email…" — should show an inline error instead of silently failing or creating a shadow student.
  7. If the teacher has Google Calendar connected, confirm 4 events appear on their calendar (best-effort — check server logs if not, since sync failures are logged, not surfaced to the UI, matching the existing single-lesson-creation behavior).
