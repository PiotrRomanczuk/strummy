AIA-2 — AI chat feedback buttons

2026-07-19 · Tranche 5 (parked) item from docs/app-blueprint/90-roadmap.md
Feature shipped Production bug found + fixed

Building AIA-2 (thumbs up/down on AI replies, writing ai_messages.is_helpful) required actually sending a real chat message through the live server and Ollama, since the feature only makes sense once a reply exists to react to. That real test immediately crashed — every single AI chat message was broken in the production build, silently, because the existing E2E tests never checked reply content. Fixing that turned out to be the bulk of this change. See "What was actually broken" below.

What was actually done

  • Feedback UI (the original ask). ChatBubble.tsx now shows a thumbs-up/thumbs-down row under any completed assistant reply. Clicking one calls submitAIFeedback(messageId, isHelpful) (new app/actions/ai-feedback.ts), which writes ai_messages.is_helpful and swaps the buttons for a "Thanks for the feedback" line. Buttons only render once the message has a persisted id — see the next point.
  • Message id plumbing. ai_messages rows didn't previously expose their id to the client at all. getLatestAssistantMessageId(conversationId) looks it up after a reply completes; useAIChat.ts's onComplete callback fetches it and attaches it to the last assistant message (ChatMessage.id, new optional field).
  • Fixed a real stale-closure bug this surfaced. onComplete originally read conversation.conversationId (React state) to know which conversation to query. On the first message of a brand-new conversation, the closure captured by useAIStream is from the render before startNewConversation() resolves and updates that state, so it was always reading a stale null — the id lookup silently never ran, and the feedback buttons never appeared for anyone's first message in a fresh chat. Fixed by threading the id through a ref (activeConvIdRef) that sendMessage writes synchronously, which onComplete always reads fresh regardless of which render's closure is executing.
  • Fixed a genuine production bug: every AI chat message crashed. See the dedicated section below — this was found while verifying AIA-2 end-to-end, not something the roadmap asked for, but it was silently breaking the core AI chat feature for every real user in the production build.

What was actually broken (found during this work, now fixed)

Root cause The chat UI creates a real AbortController and forwards its signal as an argument into generateAIResponseStream, an exported 'use server' async generator (true SSE streaming). Next.js 16.2's Server Actions runtime treats that AbortSignal as a "temporary client reference" and re-checks .aborted on it internally on every pull of the resulting stream — including pulls that happen long after the reference's valid window has closed. That throws Cannot access aborted on the server. You cannot dot into a temporary client reference from a server component. from inside Next's own wrapper code, not from anything in this codebase, and it reproduced on 100% of messages in the production build (next start).

Because the existing 16 AI E2E specs only assert that a "Strummy AI" badge renders (never the actual reply text), this crash was completely invisible to CI — the badge renders either way, whether the content is a real answer or the literal error string.

Attempt Result
Guard with if (signal?.aborted) inside the generator Still crashes — the guard itself is the dotted access.
Mirror into a server-owned AbortController before the first await, use that everywhere Still crashes — the crash is inside Next's own stream wrapper reading the original argument, independent of what the function body does with it.
Stop forwarding a live client AbortSignal into the server action at all; gate cancellation purely client-side in the consuming for await loop Fixed. No crash on any message, in any of the 17 AI E2E runs.

generateAIResponseStream no longer takes a signal parameter. Both call sites (useAIChat.ts and the admin dashboard's AIAssistantCard.tsx, which had the identical pattern) now check the local AbortSignal inside their own client-side for await loop and stop yielding chunks once it fires. The trade-off: clicking "Stop" now stops the UI from rendering further output immediately, but the server-side generation finishes in the background rather than being interrupted. There is no existing cancel/stop E2E test, so this doesn't regress any covered behavior.

Along the way, also found and fixed a second, independent issue that was masking the first one while debugging: useToolCalling in core.ts decides whether to route through OpenRouter's tool-calling SDK based solely on whether OPENROUTER_API_KEY is set — it ignores AI_PROVIDER entirely. In this dev environment both are configured, so forcing AI_PROVIDER=ollama silently still routed through OpenRouter with an Ollama-only model id (gemma3:1b), which OpenRouter correctly rejected. Not changed as part of this fix (it's a separate pre-existing provider-selection question, not a crash) — but worth flagging: AI_USE_VERCEL_SDK=false is the current workaround to force the non-tool-calling Ollama path for local testing.

How to manually test it

  1. Ensure the local Supabase tunnel is up (ssh -fN -L 54321:localhost:54321 uwh) and start the app in production mode against a real provider, e.g. Ollama: 
    npm run build
AI_PROVIDER=ollama AI_USE_VERCEL_SDK=false OLLAMA_BASE_URL=http://192.168.1.10:11434 \
  OLLAMA_DEFAULT_MODEL=gemma3:1b npx next start -p 3000
  2. Sign in as admin (p.romanczuk@gmail.com / test123_admin) and go to /dashboard/ai/chat.
  3. Send any message, e.g. "Reply with exactly one word: OK".
  4. Confirm a real reply streams in (not a crash, not literal [Cancelled]).
  5. Once streaming finishes, a thumbs-up / thumbs-down row appears under the reply (data-testid="ai-feedback-buttons"). Click thumbs-up.
  6. It should immediately switch to "Thanks for the feedback" (data-testid="ai-feedback-thanks").
  7. Verify in the DB: the corresponding row in ai_messages (role='assistant', matching conversation_id) now has is_helpful = true.
  8. Edge case worth checking: this should work on the very first message of a brand new conversation (this is exactly the stale-closure case that was broken) — not just on the second message onward.
