Dedicated Form Mockups: analysis, redesign & test coverage

2026-07-23 · branch feature/design-mockup-forms-coverage · author: Claude

What was built

The user added a first local batch of 9 "Standalone" single-file mockup exports to claude design - mockups/. Each .html is a self-contained compressed bundle (base64+gzip virtual filesystem embedding React/ReactDOM/Babel-standalone plus the mockup's own JSX), not plain static markup — reading them required extracting and decompressing the embedded {"mime","compressed","data"} blobs, or rendering them in a browser directly.

Of the 9 files, 6 re-export screens already covered by docs/app-blueprint/93-design-mockup-audit.md (Lesson List, Lesson Detail, Song Form A/B, Assignments Teacher/Student). The other 3 — Lesson Form, Student Form, and a dedicated full-page Assignment Form — are screens not in the original 44-file design-project list. All three map to real, already-routed forms.

Comparing the rendered mockups against the live app revealed the real gap wasn't implementation or tests — it was visual design. The mockups show a two-column layout with numbered sections and a live-preview sidebar; the shipped forms (including the pre-existing Song Form, used elsewhere in the codebase as the "matching SongFormA style" reference) were a single plain column. This pass closed both gaps:

Result:
  • 4 forms redesigned to the mockup's sectioned two-column + live-preview layout — Lesson Form, Student Form, Assignment Form, and Song Form (create + edit) — using new shared primitives (FormSection, FormPreviewPanel, FormAvatar, the .ed-grid-form CSS utility) so every dedicated form now looks consistent.
  • Every native field kept its exact id/name attribute — no existing E2E selector needed to change.
  • 26 new/expanded unit test cases across 4 test files, plus one stale-skipped E2E case unskipped and passing.

E2E fix

tests/e2e/teacher/lessons-crud.spec.ts's lesson CRUD lifecycle: create → view → edit → delete test was test.skip'd with the reason "Lesson create/edit forms are Coming Soon stubs — unskip when routes are built." The routes have existed for some time (proven by tests/e2e/teacher/lesson-repeat-weekly.spec.ts already exercising create successfully) — the skip reason was stale. Unskipped and verified locally against the dev Supabase stack, both before and after the layout redesign: create → view → edit → delete all pass.

Design-fidelity notes (deliberately not implemented)

  • Lesson Form: mockup's separate Date / Time / Duration / Format fields collapse into one datetime-local input; no in-person/video "format" field exists in the schema.
  • Student Form: intentionally minimal invite flow (first/last name, invite email, phone). The mockup's richer fields — skill level, avatar color, parent contact, billing rate, recurring lesson schedule — would require new DB columns + RLS work; a separate, larger initiative.
  • Assignment Form: kept single-student (mockup shows multi-student chips) — true multi-assign is a behavior change, not a layout one, so deliberately out of scope. Has a checklist editor and chord-drill editor the mockup doesn't show.

How to manually test

  1. Unit tests — from the repo root: 
    npx jest components/lessons/editorial/form \
  components/users/editorial \
  components/assignments/editorial/create \
  components/songs/editorial
    Expect all suites green.
  2. Full regression: 
    npm test
    Expect 267/267 suites passing.
  3. E2E (requires the dev server on localhost:3000 and the local Supabase stack reachable): 
    npx playwright test tests/e2e/teacher/lessons-crud.spec.ts \
  tests/e2e/teacher/lesson-repeat-weekly.spec.ts \
  tests/e2e/teacher/assignments-crud.spec.ts \
  tests/e2e/teacher/student-onboarding.spec.ts \
  tests/e2e/teacher/songs-crud.spec.ts --project="Desktop Chrome"
    Note: on a fresh next dev server, the first hit to each route/API handler pays a one-time compile cost that can exceed default timeouts — a dev-mode-only artifact, not present against a production build (CI).
  4. Manual UI walkthrough: sign in as a teacher, visit /dashboard/lessons/new, /dashboard/users/new, /dashboard/assignments/new, /dashboard/songs/new, and /dashboard/songs/[id]/edit. Confirm each renders the numbered-section two-column layout with a live preview sidebar that updates as you type/select, and that a valid submit still redirects correctly.
  5. Docs: open docs/app-blueprint/93-design-mockup-audit.md, section "Dedicated forms (2026-07-23 Standalone batch)" and the updated Song Form A/B row.
