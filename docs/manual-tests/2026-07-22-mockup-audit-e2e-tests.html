✅ Resolved — stale test credentials, not an unseeded DB. The original blocker (3 of 4 implementing agents hit invalid_credentials against loginAs()) turned out to be more specific than "the dev stack isn't seeded": playwright.config.ts hardcoded TEST_ADMIN_EMAIL/TEST_TEACHER_EMAIL/TEST_STUDENT_EMAIL to @example.com addresses that no longer exist — the shared LAN dev stack (192.168.1.75:55321) was reseeded to admin@dev.local / teacher@dev.local / student@dev.local at some point (see docs/analysis/2026-07-20-ux-frontend-audit.md) without this config catching up. Fixed in bd83e15c:
  • Repointed playwright.config.ts's defaults (and the same stale fallbacks in tests/fixtures/auth.fixture.ts and tests/helpers/seed-ids.ts) to the live @dev.local accounts.
  • admin@dev.local's password was unknown/unset — reset via the Supabase Admin API (dev-only). Verified all three roles authenticate directly against the Auth API before touching any test.
  • Fixing login exposed two previously-hidden bugs, both fixed in the same commit: inbox.spec.ts seeded notifications for a hard-coded admin UUID instead of resolving the real logged-in admin's id, and backfill-at-risk.spec.ts's seed added one stale song but getAtRiskStudents flags a student by their single most-recently-practiced song across their whole catalog — a no-op since the shared fixture student already had fresher rows. Now temporarily stales out all of that student's existing rows too, restored in afterAll.
Final verified result (real dev Supabase, Desktop Chrome, this branch's head): all 6 new/modified spec files run together — 26 passed, 14 skipped (expected — mobile-only isMobile guards on a desktop project, plus one intentionally-conditional empty-state check), 0 failed.
Separate, unrelated finding: running any @mobile/@tablet Playwright project (WebKit engine) against a plain-HTTP local dev server (localhost:3000) fails every sub-resource load with a TLS error — root cause is proxy.ts's middleware unconditionally setting Content-Security-Policy: upgrade-insecure-requests, which forces WebKit to upgrade asset requests to HTTPS even though localhost has no TLS listener. Presumably invisible in CI if CI runs against an HTTPS preview URL. Pre-existing, not caused by this branch — worth a follow-up ticket if local WebKit/mobile E2E runs are expected to work.
